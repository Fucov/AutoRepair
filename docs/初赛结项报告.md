# FeishuAutoRepair 初赛结项报告

项目名称：FeishuAutoRepair：基于 Agent 的服务自动化修复系统  
项目定位：面向研发协作场景，将运行时异常、GitHub Bug Issue、AI 诊断、代码修复、测试验证、PR Review 和飞书通知串成安全可审计闭环。  
技术栈：Python 3.12、FastAPI、pytest、Pydantic、Git worktree、GitHub API、飞书 OpenAPI、火山方舟 / Doubao、LLM Tool Use。

## 二、项目结果展示（必须包含）

### 1、总项目结果展示：

FeishuAutoRepair 最终实现的是一个“从故障发现到修复 PR 待 Review”的端到端系统，而不是一个单独的代码生成脚本。系统围绕企业研发中常见的低风险服务故障展开：当 Demo 服务发生运行时异常，或 GitHub 上出现具备证据的 Bug Issue 时，系统自动创建 Incident，关联 GitHub Issue，经过合理性检查与策略门控后创建 RepairJob，在隔离的 git worktree 中由 AI Agent 生成并验证修复，测试通过后自动创建 PR，并通过飞书卡片通知开发者 Review。

完整闭环如下：

```text
业务服务异常 / GitHub Bug Issue
→ 日志 Traceback 或 Issue 证据进入系统
→ Watcher / Issue Poller 发现问题
→ 创建或聚合 Incident
→ 创建 / 关联 GitHub Issue
→ 飞书发送异常发现卡片
→ Issue 合理性检查
→ Triage Decision + Policy Gate
→ 创建 RepairJob
→ repo lock 串行化同仓库修复
→ git worktree 创建隔离修复分支
→ RepairCase / RepairSpec / RepairSkill / ValidationPlan 构建
→ Playbook 或 Spec-Guided MiniRepairAgent 修复代码
→ 运行目标测试与全量测试
→ commit + push 修复分支
→ 创建 GitHub PR
→ 飞书发送 PR 待 Review 卡片
→ PR 合并后同步关闭 Issue 并清理 worktree / 分支
```

系统在初赛阶段聚焦 Python/FastAPI/pytest 单仓库场景，具有清晰边界：

- 不自动 merge PR。
- 不直接修改 main/master/develop。
- 所有代码修改只发生在 git worktree。
- 测试失败不创建 PR。
- 高风险安全、权限、密钥、生产数据类问题转人工。
- 外部飞书 / GitHub 配置缺失时自动降级 Mock 模式，保证本地仍可演示。

#### 2）核心部分代码展示

以下以伪代码展示项目最核心的工程链路，重点体现“故障发现、合理性检查、规格驱动修复、安全验证、PR 交付”。

##### 核心一：日志扫描与 Incident 聚合

对应实现：`autorepair/watcher.py`、`autorepair/log_parser.py`、`autorepair/incident_store.py`

```python
def scan_service_logs_once(service):
    results = []

    for log_path in service.log_paths:
        offset = get_log_offset(log_path)
        new_text, new_offset = read_new_log_text(log_path, offset)
        traceback_blocks = extract_traceback_blocks(new_text)

        for traceback in traceback_blocks:
            error_summary = extract_error_summary(traceback)
            if error_summary is None:
                continue

            incident = Incident(
                source="local_log",
                service=service.name,
                error_summary=error_summary,
                raw_traceback=traceback,
                fingerprint=error_summary.fingerprint,
            )

            final_incident, action = upsert_incident_by_fingerprint(incident)

            if action == "created":
                append_audit_event("incident_created", final_incident)
            else:
                append_audit_event("incident_updated", final_incident.occurrence_count)

            results.append((final_incident, action))

        set_log_offset(log_path, new_offset)

    return results
```

该模块解决两个基础问题：第一，系统能够从日志中自动发现 Traceback；第二，相同错误不会重复刷屏，而是通过 fingerprint 聚合为同一个 Incident，并更新 occurrence_count。

##### 核心二：GitHub Issue 合理性检查

对应实现：`autorepair/issue_validator.py`

```python
def validate_bug_issue(issue):
    labels = set(issue.labels)
    title = issue.title
    body = issue.body

    if "bug" not in labels and "[Bug]" not in title:
        return invalid("缺少 bug 标记")

    if issue_has_status(labels, ["autorepair:closed", "autorepair:repairing", "autorepair:pr-ready"]):
        return invalid("该 Issue 已处理或处理中")

    evidence = {
        "service_or_module": has_service_or_module(title, body),
        "reproduction_steps": has_reproduction_steps(body),
        "expected_behavior": has_expected_behavior(body),
        "actual_behavior": has_actual_behavior(body),
        "error_or_traceback": has_error_or_traceback(body),
        "failing_test": has_failing_test_or_command(body),
    }

    if contains_high_risk_keywords(title + body):
        return invalid("涉及安全、权限、密钥或生产数据，转人工")

    if not evidence["reproduction_steps"] and not evidence["error_or_traceback"]:
        return invalid("缺少复现步骤和错误证据")

    if count_true(evidence) < 3:
        return invalid("证据不足，需要补充信息")

    return valid("证据足够，允许进入 triage")
```

该模块借鉴 ClawSweeper 的保守治理思想：自动化系统不能因为“看起来像 Bug”就进入自动修复，必须先判断证据完整度、风险和当前处理状态。

##### 核心三：Triage Decision 与 Policy Gate

对应实现：`autorepair/agent/triage/`、`autorepair/agent/triage/policy_gate.py`

```python
def run_triage_and_policy_gate(incident, issue):
    evidence = collect_evidence(incident, issue)
    decision = llm_triage_as_json(evidence)
    decision = Decision.model_validate(decision)

    gate = should_auto_fix(decision)
    write_diagnostic_report(incident, decision, gate)

    if not gate.allowed:
        send_manual_intervention_card(
            reason=gate.blocked_reason,
            evidence=evidence,
        )
        return "human_required"

    create_repair_job(
        incident_id=incident.id,
        issue_number=issue.number,
        policy_decision=decision,
        risk_level=decision.risk_level,
    )
    send_repair_plan_ready_card(decision)
    return "queued"


def should_auto_fix(decision):
    if decision.decision != "auto_fix":
        return blocked("decision is not auto_fix")
    if decision.confidence != "high":
        return blocked("confidence is not high")
    if not decision.evidence:
        return blocked("missing evidence")
    if not decision.fix_plan:
        return blocked("missing fix plan")
    if decision.requires_human_approval:
        return blocked("requires human approval")
    if contains_high_risk(decision.risks):
        return blocked("high risk")
    return allowed()
```

这里 AI 输出并不直接决定是否修复，工程层 Policy Gate 会再次把关。这样既使用 AI 的归纳能力，也避免模型误判导致自动化越权。

##### 核心四：RepairCase 限定修复任务和可修改文件

对应实现：`autorepair/repair_agent/repair_case.py`

```python
def build_repair_case(context, issue=None):
    text = join(
        context.error_type,
        context.error_message,
        context.raw_traceback,
        issue.body if issue else "",
    )

    scenario_id = infer_scenario_id(text)

    allowed_files = {
        "user-missing-profile": ["demo_service/service.py"],
        "order-zero-division": ["demo_service/order_service.py"],
        "ticket-timezone-sla": ["demo_service/ticket_service.py"],
        "ticket-idempotency-duplicate": ["demo_service/ticket_service.py"],
        "ticket-nameerror-overdue": ["demo_service/ticket_service.py"],
    }.get(scenario_id, [])

    target_tests = map_scenario_to_target_tests(scenario_id)

    confidence = 0.9 if allowed_files else 0.3

    return RepairCase(
        incident_id=context.incident_id,
        scenario_id=scenario_id,
        allowed_files=allowed_files,
        forbidden_files=[".env", ".git", "pyproject.toml", "requirements.txt", "tests"],
        target_tests=target_tests,
        expected_behavior=map_expected_behavior(scenario_id),
        current_failure=f"{context.error_type}: {context.error_message}",
        confidence=confidence,
    )
```

RepairCase 的意义是把“要修什么、允许修哪里、用什么测试验收”显式化。它阻止 LLM 随意选择文件，也解决了 Traceback 指到路由层但真正问题在 service 层时容易误修的问题。

##### 核心五：RepairSpec 规格驱动修复

对应实现：`autorepair/repair_agent/spec_builder.py`

```python
def build_repair_spec(case, context):
    if case.scenario_id == "ticket-timezone-sla":
        return RepairSpec(
            function_under_repair="submit_ticket",
            caller_expectation="submit_ticket 接受带时区的 sla_deadline 并返回创建的工单",
            preconditions=[
                "sla_deadline 是 ISO 8601 格式字符串",
                "sla_deadline 可能带时区，也可能不带时区",
            ],
            postconditions=[
                "fromisoformat 结果必须被统一为 timezone-aware datetime",
                "naive/aware 时间比较必须统一到 UTC aware",
                "过期工单返回 overdue",
                "未过期工单返回 open",
            ],
            invariants=[
                "不得使用 datetime.utcnow() 与 aware datetime 比较",
                "所有 datetime 比较必须在同一时区基础上进行",
            ],
            violation="naive datetime 与 aware datetime 比较导致 TypeError",
            acceptance_tests=case.target_tests,
        )

    return fallback_spec(case, context)
```

这一层借鉴 FM-Agent 的函数规格思想：先从调用方期望、测试和 Issue 证据中构建规格，再让 Agent 根据规格修复。目标不是简单“让报错消失”，而是满足预期行为和不变量。

##### 核心六：RepairSkill 路由

对应实现：`autorepair/repair_agent/skills/`

```python
def select_repair_skills(case, spec):
    skills = []

    if contains(case.current_failure, ["timezone", "offset-naive", "offset-aware"]):
        skills.append(DateTimeTimezoneSkill())

    if contains(case.current_failure, ["NoneType", "not subscriptable", "missing user"]):
        skills.append(NullGuardSkill())

    if contains(case.current_failure, ["ZeroDivisionError"]):
        skills.append(ZeroDivisionSkill())

    if contains(spec.violation, ["idempotency_key", "duplicate", "same ticket_id"]):
        skills.append(IdempotencySkill())

    if contains(case.current_failure, ["NameError"]):
        skills.append(NameErrorSkill())

    return skills
```

RepairSkill 将组织修复经验结构化。例如 DateTimeTimezoneSkill 会提示模型使用 `datetime.now(timezone.utc)`，并将 `fromisoformat` 得到的 datetime 统一转换为 UTC aware。

##### 核心七：ValidationPlan 与测试门禁

对应实现：`autorepair/repair_agent/validator.py`

```python
def run_validation_plan(tools, plan, phase, spec):
    if phase == "before":
        result = tools.run_tests(plan.reproduce_command or plan.target_commands[0])
        if result.ok:
            return not_reproducible()
        return reproduced_failure(
            failed_command=result.command,
            violated_spec_item=find_violated_spec(spec),
        )

    if phase == "after":
        for command in plan.target_commands:
            result = tools.run_tests(command)
            if not result.ok:
                return failed("target_test_failed", command, find_violated_spec(spec))

        for command in plan.related_commands:
            result = tools.run_tests(command)
            if not result.ok:
                return failed("related_test_failed", command, "回归测试失败")

        full = tools.run_tests(plan.full_command)
        if not full.ok:
            return failed("full_test_failed", plan.full_command, "全量测试失败")

        return passed()
```

ValidationPlan 确保修复前能复现、修复后目标测试和全量测试都通过。测试失败时，系统会将失败归因到 RepairSpec 的 postcondition / invariant，而不是只把原始 stderr 交给模型。

##### 核心八：Spec-Guided MiniRepairAgent

对应实现：`autorepair/repair_agent/loop.py`、`autorepair/repair_agent/tools.py`

```python
def run_spec_guided(context, repair_case, repair_spec, skills, validation_plan, history_context):
    tools = MiniRepairTools(
        context.worktree_path,
        allowed_files=set(repair_case.allowed_files),
        forbidden_files=set(repair_case.forbidden_files),
    )

    before = run_validation_plan(tools, validation_plan, "before", repair_spec)
    if before.target_ok:
        return RepairAgentResult(status="not_reproducible")

    prompt = build_initial_prompt(
        context=context,
        repair_case=repair_case,
        repair_spec=repair_spec,
        selected_skills=skills,
        validation_plan=validation_plan,
        history_context=history_context,
    )

    for phase in ["UNDERSTAND", "PLAN", "EDIT", "VALIDATE", "FINALIZE"]:
        for step in phase_budget[phase]:
            call = llm.chat_json(prompt)

            if call.tool in ["apply_replace", "rewrite_file"]:
                if call.args["path"] not in repair_case.allowed_files:
                    prompt.add("只能修改 allowed_files")
                    continue

            if phase == "EDIT":
                checkpoint = save_checkpoint(repair_case.allowed_files)

            result = tools.dispatch(call)

            if result.changed:
                diff = tools.git_diff()
                prompt.add(diff)

            if phase == "VALIDATE" and result.failed:
                feedback = build_spec_violation_feedback(
                    failed_command=result.command,
                    violated_spec_item=find_violated_spec(repair_spec),
                    current_diff=tools.git_diff(),
                )
                prompt.add(feedback)

                if consecutive_no_improve:
                    rollback(checkpoint)

    after = run_validation_plan(tools, validation_plan, "after", repair_spec)
    if after.target_ok and after.full_ok:
        return fixed(diff=tools.git_diff())
    return test_failed_or_needs_human(after.failure_summary)
```

Agent 采用阶段式预算，而不是无限循环：UNDERSTAND 读代码，PLAN 输出计划，EDIT 修改 allowed_files，VALIDATE 运行测试并根据规格反馈修正，FINALIZE 输出结果。这样可以控制模型成本和风险。

##### 核心九：Repair Executor 生成 PR

对应实现：`autorepair/repair/executor.py`

```python
def execute_next_repair_job():
    job = find_oldest_queued_job()
    if not job:
        return ok("No queued repair job")

    if has_active_job_or_open_pr(job.issue_number):
        return ok("Already processing")

    with acquire_repo_lock(job.repo):
        update_job(job, status="running")
        create_repair_worktree(job.repo_path, job.base_branch, job.repair_branch)

        context = collect_repair_context(job)
        agent_context = build_repair_agent_context(job, incident)
        case = build_repair_case(agent_context)
        spec = build_repair_spec(case, agent_context)
        skills = select_repair_skills(case, spec)
        plan = build_validation_plan(case, spec, agent_context)
        history = collect_history_context(job.worktree_path, case.allowed_files)

        result = try_apply_skill_backed_playbook(case, spec, skills, plan)
        if result is None:
            result = MiniRepairAgent(LLMClient()).run_spec_guided(
                agent_context, case, spec, skills, plan, history
            )

        if result.status != "fixed":
            update_job(job, status="test_failed_or_human_required")
            comment_issue(job.issue_number, result.summary)
            send_manual_intervention_card(result.summary)
            return failed(result.summary)

        diff = get_git_diff(job.worktree_path)
        commit_sha = git_commit_all(job.worktree_path, build_commit_message(job, result))
        git_push_branch(job.worktree_path, job.repair_branch)
        pr = create_pull_request(build_pr_title(result), build_pr_body(job, result, diff), job.repair_branch, job.base_branch)

        update_job(job, status="pr_created", pr_number=pr.number, pr_url=pr.html_url)
        replace_autorepair_status_label(job.issue_number, "autorepair:pr-ready")
        send_fix_pr_ready_card(pr_url=pr.html_url, fix_summary=result.summary)
        return success(job)
```

Executor 是自动修复的总控层。它不让 Agent 直接 commit、push 或创建 PR，而是在测试通过后统一执行交付动作，保证职责隔离。

#### 3）项目亮点介绍

##### 亮点一：完整闭环，覆盖真实研发协作流程

FeishuAutoRepair 覆盖从故障发现到 PR Review 的完整链路，而不是只做“模型给出一段修复建议”。系统包含：

- 业务 Demo 服务。
- 日志 Traceback 采集。
- Incident 去重与聚合。
- GitHub Issue 统一事实源。
- 飞书卡片通知。
- Triage / Policy Gate。
- RepairJob 队列。
- git worktree 隔离修复。
- Repo lock 并发控制。
- Agent 工具调用修复。
- pytest 验证。
- 自动创建 PR。
- PR 合并状态同步。
- 审计日志和 transcript。

这条链路能够落地到研发团队，因为它保留了团队熟悉的 Issue、PR、Review 和飞书通知机制。

##### 亮点二：借鉴 ClawSweeper，对 Bug 合理性进行考察

项目没有让 AI 对所有 Bug Issue 自动修复，而是在进入修复前增加“合理性检查 + 保守策略门控”。这一点借鉴了 ClawSweeper 的维护机器人思想：自动化可以帮助处理大量问题，但只有证据强、范围明确、风险可控的问题才适合自动动作。

在 FeishuAutoRepair 中，合理性检查包括：

- 是否有 bug 标签或 `[Bug]` 标题。
- 是否已处于 repairing、pr-ready、closed 等状态。
- 是否包含服务 / 模块信息。
- 是否包含复现步骤。
- 是否包含期望行为和实际行为。
- 是否包含错误信息、Traceback 或失败测试。
- 是否包含安全、密钥、权限、生产数据库等高风险内容。

结果分为：

- 证据足够：进入 triage。
- 信息不足：评论 needs-info，等待补充。
- 高风险：标记 human-required，发送人工介入卡。
- 已处理：跳过，避免重复处理。

这个设计让项目从“AI 自动修复玩具”变成“受控自动化系统”。

##### 亮点三：借鉴 FM-Agent，对修复代码合理性进行考察

项目借鉴 FM-Agent 的核心思路：修复前先构造函数/调用规格，再验证修复是否满足规格。传统 Traceback-driven Agent 往往看到哪里报错就改哪里，可能把路由层、测试层或错误位置当成真实根因。FeishuAutoRepair 通过 RepairCase 和 RepairSpec 将修复任务显式化：

- RepairCase 判断问题类型、目标入口、allowed_files、forbidden_files、目标测试。
- RepairSpec 描述调用方期望、前置条件、后置条件、不变量、当前违反项。
- ValidationPlan 将测试失败对应到 RepairSpec 的规格条目。
- Prompt 明确告诉模型目标是满足 RepairSpec，而不是单纯让异常消失。

例如 timezone Bug 的合理修复不是“try/except 掉 TypeError”，而是统一 datetime 时区语义，保证带时区和不带时区的 SLA deadline 都能按照业务规则返回 open / overdue。

##### 亮点四：Playbook + Agent 的双层修复架构

项目并不完全依赖 LLM 即兴发挥。对高频、低风险、模式稳定的问题，系统优先使用确定性 Playbook：

- DateTime timezone。
- NameError 字符串字面量。
- UnboundLocalError import scope。
- ZeroDivision 简单防护。
- Null guard 简单防护。

如果 Playbook 不命中或测试失败，再进入 Spec-Guided MiniRepairAgent。这样既提升演示稳定性，也保留对未知问题的扩展能力。Playbook 是沉淀组织经验，Agent 是处理开放问题，两者都受 RepairCase / RepairSpec / ValidationPlan 约束。

##### 亮点五：安全边界清晰

项目始终坚持“自动修复不等于自动上线”：

- 不自动 merge。
- 不直接改主分支。
- 不改 `.env`、`.git`、token、secret 等敏感文件。
- 默认不改测试文件。
- 只运行 pytest。
- 只修改 allowed_files。
- 测试失败不创建 PR。
- 所有 PR 需要人类 Review。

该安全边界符合企业真实落地要求。

##### 亮点六：飞书协作特色明确

项目不是简单把日志发到飞书，而是将飞书作为研发协作入口：

- IncidentDetectedCard：首次发现故障。
- RepairPlanReadyCard：诊断完成，准备修复。
- FixPrReadyCard：PR 已生成，请 Review。
- ManualInterventionCard：需要人工介入。
- PeriodicDigestCard：周期摘要。

卡片只展示决策所需摘要，详细 Traceback、fingerprint、测试命令和 diff 放到报告、Issue 或 PR 中，避免移动端信息过载。

#### 4）AI 亮点介绍

此模块重点体现项目的 AI 工程化深度与落地价值。

##### 项目中使用了哪些高阶 AI 技巧？

1. **结构化 JSON 输出**  
   Triage Decision、修复计划和工具调用都要求 JSON 输出，再通过 Pydantic 进行 Schema 校验。这样避免模型输出自然语言导致系统无法执行。

2. **LLM + Policy Gate 双重判断**  
   LLM 负责归纳根因和修复建议，Policy Gate 负责工程风险控制。模型输出不是最终命令，必须经过规则放行。

3. **Spec-Guided Repair**  
   通过 RepairSpec 将调用方期望、preconditions、postconditions、invariants 显式注入 Prompt。模型修复的目标是满足规格，而不是简单消除异常。

4. **Tool Use Agent**  
   MiniRepairAgent 不直接获得任意 shell 权限，只能调用 read_file、get_file_excerpt、search_text、run_tests、apply_replace、rewrite_file、git_diff、finish 等受限工具。

5. **RepairSkill 路由**  
   根据错误类型选择 DateTimeTimezoneSkill、NullGuardSkill、ZeroDivisionSkill、IdempotencySkill、NameErrorSkill、ImportScopeSkill，为模型注入针对性修复策略。

6. **Validation Feedback**  
   测试失败后，系统不只把 stderr 原样返给模型，而是生成 SpecViolationFeedback，指出失败命令、当前 diff、剩余预算、违反的规格条目。

7. **Playbook 兜底**  
   对确定性错误先使用规则修复，降低模型不稳定性；复杂问题再交给 Agent。

8. **HistoryContext 增强**  
   读取 git log / blame 附近上下文，让模型理解近期修改历史，降低误改概率。

9. **Transcript 审计**  
   保存每一步 ToolCall 与 ToolResult，方便复盘模型为什么这么改、测试在哪里失败、是否越界。

##### 项目中人和 AI 的分工是怎么样的？

AI 负责：

- 根据日志、Issue、Traceback、测试输出整理证据。
- 输出结构化诊断结果。
- 构造或使用 RepairSpec。
- 在 allowed_files 内读取代码、搜索函数、提出最小修改。
- 运行测试，根据失败反馈迭代。
- 生成修复摘要、PR 描述、飞书卡片文案。

人负责：

- 配置飞书、GitHub、Ark 等系统边界。
- 决定哪些仓库和服务允许接入。
- Review AI 创建的 PR。
- 对 high-risk、needs-info、test_failed、human-required 问题进行人工处理。
- 最终决定是否 merge。

这种分工让 AI 承担重复、低风险、可验证的工作，人类保留最终判断权。

##### 项目中包含哪些核心模型选型思路？

项目面向火山方舟 / Doubao 设计模型路由：

- **代码修复模型**：用于代码理解、ToolCall 决策、RepairPlan 生成和复杂修复。
- **摘要模型**：用于飞书卡片、Issue 评论、失败报告、PR 描述等低成本文本生成。

选型思路不是“越强越好”，而是按任务拆分：

- 复杂代码修复需要更强推理能力。
- 卡片摘要和报告生成更看重稳定、便宜、快速。
- 所有模型输出都要经过 JSON 校验、Policy Gate、测试验证和人工 Review。

##### 引入 AI 后对原有工作流带来哪些改变？

传统工作流：

```text
人发现异常
→ 人复制日志
→ 人创建 Issue
→ 人判断是否可修
→ 人定位代码
→ 人修改代码
→ 人跑测试
→ 人创建 PR
→ 人通知 Review
```

引入 FeishuAutoRepair 后：

```text
系统发现异常
→ 自动创建 Incident / Issue
→ AI 诊断和规格化
→ 系统判断风险
→ AI 在 worktree 中修复
→ 系统运行测试
→ 系统创建 PR
→ 飞书通知人类 Review
```

人类从重复性定位、复制信息、机械修复中释放出来，转向 Review、决策和处理复杂问题。AI 不替代研发流程，而是嵌入研发流程。

##### AI 工程化落地价值

项目的 AI 亮点不在于“模型能生成代码”这一点本身，而在于模型被放进了工程闭环：

- 输入有证据收集和风险过滤。
- 输出有 Schema 校验。
- 行为有工具权限限制。
- 修改有 allowed_files 限制。
- 结果有测试验证。
- 交付有 PR Review。
- 过程有审计记录。
- 失败有人工介入报告。

这套设计可以推广到更多企业内部 Agent 场景。

## 三、其他信息（可选）-自由发挥区

### 项目可推广价值

FeishuAutoRepair 不只适用于比赛 Demo，也可以推广到企业内部研发平台：

- 接入测试环境服务日志，实现低风险 runtime bug 自动转 Issue。
- 接入 GitHub / GitLab Issue，实现 Bug 报告自动分诊。
- 接入 CI 失败日志，实现测试失败自动定位和修复 PR 生成。
- 接入飞书群、飞书多维表格或知识库，实现修复记录沉淀。
- 将 RepairSkill 和 Playbook 扩展为组织级故障修复知识库。

### 项目设计取舍

本项目最重要的取舍是：**自动修复不等于自动上线**。系统最终交付的是可审阅 PR，而不是直接修改生产代码。这样既能展示 AI 自动修复能力，又能保持企业研发流程中的人类审核和责任边界。

### 后续优化方向

1. 补齐更多 RepairSkill 和 Playbook，提升多场景修复成功率。
2. 将 RepairSpec 从规则模板升级为“规则 + LLM 生成 + Schema 校验”的混合模式。
3. 增强 Dashboard 的时间线、diff 预览、transcript 查看能力。
4. 完善飞书多维表格写入，把 Incident、RepairJob、PR 作为工单台账沉淀。
5. 支持 GitHub Actions / pytest 失败日志作为触发源。
6. 增加修复成功率、平均耗时、模型成本、人工介入率等指标。

### 答辩总结建议

可以用三句话概括项目价值：

1. **流程闭环**：从异常发现到 PR Review 通知形成完整自动化链路。
2. **可信自治**：借鉴 ClawSweeper 的保守治理，只让低风险、有证据、可测试的问题进入自动修复。
3. **规格驱动**：借鉴 FM-Agent 的规格推导思想，让 Agent 修复业务正确性，而不是只消除局部报错。
