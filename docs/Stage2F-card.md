你这个判断是对的：**最终给用户看的核心不是 demo 网页，而是飞书里的“通知卡片体验”**。
所以卡片一定不能是那种“默认大图 + 一堆文案 + 一个按钮”的粗糙样式，而应该做成：

* **状态明确**
* **重点信息一眼看懂**
* **能直接跳到下一步动作**
* **不同场景分卡设计**
* **尽量少打扰，但关键时刻必须提醒**

我建议你不要只做 1 张万能卡，而是做 **5 张核心卡 + 1 张可选卡**。这样评委会明显感受到你的系统是“有产品设计思维”的。

---

# 一、先给结论：建议保留的卡片类型

## 核心 5 张卡

1. **故障发现 / 已受理告警卡**
2. **自动修复完成 / PR 待 Review 卡**
3. **自动修复失败 / 需人工介入卡**
4. **重复故障升级 / 风险升级卡**
5. **周期性总结卡（日报 / 周报）**

## 可选第 6 张

6. **修复已合并 / 闭环成功卡**

---

# 二、总设计原则

先说结论：**不是每张卡都要放图。**

你现在那个“大横幅图 + 大字 AI 自动修 Bug”的问题在于：

* 信息密度低
* 太像宣传海报，不像工作通知
* 占空间，用户还得下滑
* 真正重要的信息（哪个服务、什么 bug、影响多大、下一步做什么）反而不突出

---

## 更合理的设计原则

### 1）卡片结构统一

所有卡片都统一成：

```text
[状态标题栏]
[一句话结论]
[关键信息区]
[说明区 / 风险区 / 建议区]
[按钮区]
```

---

### 2）颜色分状态

* **红色**：告警 / 故障 / 高风险
* **橙色**：升级 / 需关注 / 风险增加
* **蓝色**：处理中 / 已受理 / 分析中
* **绿色**：修复成功 / 已闭环
* **灰色**：汇总 / 周报 / 信息型通知

---

### 3）按钮不要太多

每张卡 **2~4 个按钮最合适**。
太多按钮会显得乱，也不利于飞书内查看。

建议按钮优先级：

* 主按钮：**查看详情**
* 次按钮：**查看日志 / 查看 Issue / 查看 PR**
* 辅助按钮：**查看审计记录 / 查看诊断报告**

如果你当前没有回调体系，不建议做“一键通过”“一键驳回”这种假交互。
更稳妥的是跳转到：

* GitHub PR
* GitHub Issue
* 本地/内部详情页
* 审计页面 / Markdown 报告页

---

### 4）图片使用建议

不是每张卡都放图。

## 建议：

* **故障告警卡**：不要大图，只用小图标/状态图标
* **修复完成待 Review 卡**：可放一张简洁 Banner 图
* **周期性总结卡**：可以放一张小型趋势图或摘要图
* **失败/人工介入卡**：不要大图，突出“问题和建议”
* **升级卡**：不用图，用红橙色提示条就够了
* **闭环卡**：可选小成功插图，但不是必须

# 1）故障发现 / 已受理告警卡

这是最基础、最重要的一张卡。

## 使用场景

当系统首次发现一个新的 Incident 时发送。

## 目标

告诉用户：

* 出故障了
* Agent 已经接收到
* 目前在哪个阶段
* 影响范围和严重性如何
* 可以点哪里查看详情

---

## 卡片内容建议

### 标题

**【高优先级】检测到服务异常，AutoRepair 已受理**

或者：

* 【P0】检测到生产故障，已进入自动诊断
* 【P1】发现服务异常，已生成 Incident

---

### 一句话摘要

**Acme SupportDesk Lite 在 10:08 发生运行时异常，Agent 已完成接收并开始诊断。**

---

### 关键信息区（建议 2 列）

* 服务名称：Acme SupportDesk Lite
* 环境：Local Demo / Test / Prod
* Incident ID：INC-20260425-xxxx
* 严重级别：P0 / P1 / P2
* 错误类型：TypeError / ZeroDivisionError / RateLimitError
* 触发入口：local_log / github_issue
* 发生时间：2026-04-25 10:08:32
* 当前阶段：已受理 / 诊断中

---

### 错误摘要区

* 错误位置：`demo_service/ticket_service.py:35`
* 错误摘要：`can't compare offset-naive and offset-aware datetimes`
* 最近出现次数：`6 次`
* 是否重复问题：是 / 否

---

### 建议说明区

可以加一段小字：

**系统将自动进行日志分析、代码定位、测试验证。若判断为可自动修复问题，将生成修复分支与 PR。**

---

### 按钮

* **查看 Incident 详情**
* **查看 Traceback**
* **查看 GitHub Issue**（如果有）
* **查看审计记录**

---

### 图片建议

**不要放大图。**
只需要一个小状态图标即可，例如：

* 红色警报圆点
* 服务告警图标
* 小型系统异常插图（可选）

---

## 这张卡的 prompt（给 AI 生成设计 / 代码）

```text

设计要求：
1. 顶部使用红色或橙红色状态标题栏，标题为“【高优先级】检测到服务异常，AutoRepair 已受理”。

3. 中间使用两列信息块展示：
   - 服务名称
   - 环境
   - Incident ID
   - 严重级别
   - 错误类型
   - 触发来源
   - 发生时间
   - 当前阶段
4. 增加“错误摘要”区域，展示：
   - 错误位置
   - 错误摘要
   - occurrence_count
   - 是否重复问题
5. 增加一段简短说明：系统将继续进行日志分析、代码定位、测试验证，并在可自动修复时生成修复方案。
6. 底部放 3~4 个按钮：
   - 查看 Incident 详情（主按钮）
   - 查看 Traceback
   - 查看 GitHub Issue（如果存在）
   - 查看审计记录
7. 不要使用夸张 AI 宣传图，不要大字海报风。
8. 卡片整体要像企业告警通知，而不是营销海报。
```

# 2）自动修复完成 / PR 待 Review 卡

这是最适合展示“成果感”的一张卡。

## 使用场景

当 Agent 已经：

* 完成分析
* 生成补丁
* 跑过测试
* 创建了 PR

然后发给用户 Review。

---

## 目标

告诉用户：

* Bug 已自动修
* 修复是怎么做的
* 改了哪些文件
* 测试结果如何
* 现在请你 Review PR

---

## 卡片内容建议

### 标题

**【已修复】AutoRepair 已完成补丁生成，PR 待 Review**

---

### 一句话摘要

**系统已针对本次异常完成根因定位、代码修复和测试验证，并生成 Pull Request，请 Review。**

---

### 关键信息区

* 服务名称
* Incident ID
* PR 编号 / 标题
* 修复分支：`fix/inc-20260425-xxxx`
* 修复类型：运行时异常 / 业务逻辑缺陷
* 风险等级：Low / Medium
* 当前阶段：PR 待 Review

---

### 修复摘要区

* 根因：例如“带时区时间与无时区时间比较导致 TypeError”
* 修复策略：例如“统一转换为 timezone-aware datetime 后再比较”
* 修改文件：

  * `demo_service/ticket_service.py`
  * `demo_service/tests/test_ticket_contract.py`
* 关键测试：

  * `pytest -q`
  * `pytest -q -m agent_target`
* 测试结果：通过 18/18

---

### 按钮

* **查看 PR 详情**（主按钮）
* **查看变更 Diff**
* **查看测试报告**
* **查看修复审计记录**

---

### 图片建议

这张卡 **可以放图**，但不要像你当前那个“AI 自动修 Bug”大海报。

推荐图像风格：

* 深蓝 / 蓝紫渐变
* 极简科技感
* 左侧少量文案
* 右侧抽象网格/波纹/代码流线
* 不要大字盖满画面
* 图中只放一句核心视觉文案即可：

例如：

* “AutoRepair 已生成修复方案”
* “Bug Fixed · PR Ready for Review”

---

## 这张卡的 prompt

```text
请为“AutoRepair Agent 自动修复完成，PR 待 Review”场景设计一张飞书交互卡片。风格应专业、可信、偏工程效率工具，而非营销海报风。

设计要求：
1. 顶部使用绿色状态标题栏，标题为“【已修复】AutoRepair 已完成补丁生成，PR 待 Review”。
2. 标题下方用一句话说明：系统已完成根因分析、代码修复、测试验证，并生成 Pull Request。
3. 中间展示修复关键信息：
   - 服务名称
   - Incident ID
   - PR 编号 / 标题
   - 修复分支
   - 修复类型
   - 风险等级
   - 当前阶段
4. 增加“修复摘要”区域：
   - 根因分析
   - 修复策略
   - 修改文件列表
   - 测试命令
   - 测试结果
5. 底部放 4 个按钮：
   - 查看 PR 详情（主按钮）
   - 查看变更 Diff
   - 查看测试报告
   - 查看修复审计记录
6. 可以放一张简洁横幅图，但必须克制，建议是深蓝到紫色渐变 + 抽象网格/波纹的科技风图，不要大面积夸张宣传字。
7. 图片中的文案建议只保留一句：“Bug Fixed · PR Ready for Review” 或 “AutoRepair 已生成修复方案”。
8. 卡片要突出“已修复、可 Review、测试通过”三件事。
```

# 3）自动修复失败 / 需人工介入卡

这是你系统“可信”的关键。
因为不是所有 bug 都能修，你必须敢于告诉用户：**这个问题不适合自动修。**

---

## 使用场景

当出现以下情况：

* 依赖缺失
* 外部服务故障
* 环境问题
* 无法复现
* 自动修复失败
* 多轮尝试后仍未通过测试

---

## 目标

告诉用户：

* 系统没有盲目乱修
* 问题在哪一层
* 为什么需要人来处理
* 你可以看哪些证据

---

## 标题

**【需人工介入】问题未自动修复，请处理**

---

## 一句话摘要

**系统已完成诊断与尝试，但当前问题不适合自动修复，建议人工排查。**

---

## 关键信息区

* 服务名称
* Incident ID
* 当前阶段：人工介入
* 失败原因分类：

  * 依赖缺失
  * 外部服务异常
  * 环境配置问题
  * 测试未通过
  * 风险过高
* 最近尝试次数
* 诊断结论

---

## 详细说明区

* 根因判断
* 自动修复失败原因
* 已执行动作
* 建议人工排查方向

例如：

* 缺少依赖包 `xxx`
* GitHub 分支保护限制 PR 自动提交
* API 限流 / 外部接口返回 429
* 修改后测试仍未通过
* 当前变更风险较高，未继续自动提交

---

## 按钮

* **查看诊断报告**
* **查看错误日志**
* **查看已尝试修复记录**
* **查看相关 Issue**

---

## 图片建议

**不要放大图。**
用一个简洁的黄/橙色风险提示视觉即可。

---

## 这张卡的 prompt

```text

设计要求：
1. 顶部使用橙色或红橙色标题栏，标题为“【需人工介入】问题未自动修复，请处理”。
2. 用一句话说明：系统已完成诊断和自动修复尝试，但当前问题不适合自动处理。
3. 展示关键信息：
   - 服务名称
   - Incident ID
   - 当前阶段
   - 失败原因分类
   - 最近尝试次数
   - 诊断结论
4. 增加“已执行动作”区域：
   - 是否完成日志分析
   - 是否完成代码定位
   - 是否尝试修复
   - 是否运行测试
5. 增加“建议人工排查方向”区域。
6. 底部按钮：
   - 查看诊断报告（主按钮）
   - 查看错误日志
   - 查看尝试记录
   - 查看相关 Issue
7. 不要使用宣传图，不要使用夸张文案。
8. 卡片要突出“为什么没有自动修复”。
```

* 服务名称
* occurrence_count

* 最近发生时间
* 升级原因

## 说明区

* 是否已有修复 PR

## 按钮

* **查看 Incident 详情**

PR 已被 merge，故障闭环。

---
