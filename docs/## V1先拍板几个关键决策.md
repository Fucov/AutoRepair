## V1先拍板几个关键决策

**1. 飞书侧：用 API，不用 CLI 作为主链路。**
CLI 更适合开发者本地调试、插件/小程序工程流程；你这个项目需要的是“服务端自动发卡片、写多维表格、更新状态、可选接收卡片按钮动作”，所以应该走 **Feishu OpenAPI + 应用机器人**。飞书服务端 API 支持发送消息，卡片可以配置交互，多维表格也有新增记录接口；这正好对应“通知 + 状态看板 + 审计记录”的核心链路。([飞书开放平台][1])

**2. LLM 侧：主用火山方舟 Ark / Doubao，不需要再引入其他模型。**
你已经有 Doubao 2.0 和 1.6 的无限量额度，这对初赛非常关键。建议：**Doubao 2.0 负责代码分析和 patch 生成，Doubao 1.6 负责摘要、卡片文案、失败原因解释**。火山方舟官方文档显示方舟支持 Chat API、Responses API、Bot API、Endpoint 管理、API Key 等能力，Python SDK 也有安装入口。([火山引擎][2])

**3. GitHub 侧：必须新建修复分支 + PR，不自动 merge。**
每个 incident 建一个分支，例如 `fix/inc-20260424-001-null-user-id`。Agent 可以 commit、push、create PR，但不要 merge。GitHub REST API 创建 PR 需要 `head`、`base`、`title` 等参数，merge PR 是另一个独立接口，所以工程上完全可以把“创建 PR”和“合并 PR”隔离开。([GitHub Docs][3])

**4. 初赛本地部署：需要一个 demo 故障服务，而且必须做。**
比赛要求展示“从报错到修复通知”的全过程。如果只接真实项目，演示不可控；你应该写一个 **demo Web 服务 + seed bug 脚本**，专门制造 `AttributeError / KeyError / TypeError / pytest fail`。Python 3.12 没问题，Python 官方 `traceback` 模块本来就用于提取、格式化、打印 Python 栈信息。([Python documentation][4])

# 项目最终定义

项目名称可以定为：

> **Feishu AutoRepair Agent：面向研发协作的飞书内闭环自动修复 Agent**

一句话定义：

> 当本地 Web 服务发生异常或 GitHub Issue/测试失败出现时，系统自动捕获 traceback，调用 Doubao 分析根因，在 Git worktree 沙箱中生成并验证补丁，通过 GitHub 创建修复 PR，并将全过程同步到飞书卡片与多维表格，开发者只需要 Review PR。

这个定义的好处是：**不是做一个“自动上线系统”，而是做一个“自动生成可审阅修复 PR 的协作 Agent”**。更安全、更落地，也更符合比赛周期。

# 关于“回调”的准确理解

你现在的理解基本对，但要区分三件事：

## 1. 没有回调也能完成题目基础要求

题目说的是：

> Agent 自动修复并提交 PR，最后通过飞书卡片通知开发者 Review。

这意味着最小闭环可以是：

```text
服务报错
→ Agent 自动分析
→ Agent 自动修复
→ Agent 自动跑测试
→ Agent 自动创建 PR
→ 飞书通知“已修复，请 Review”
```

这里不需要用户点按钮。

## 2. 回调的意义不是“必须点击后才能 commit”

更推荐的设计是：**Agent 自己创建 commit 和 PR，用户在 GitHub PR 里 Review**。飞书回调按钮不是必须用来控制 commit，而是用来做增强交互，例如：

```text
查看 PR
重新运行测试
转人工处理
生成 RCA
忽略该问题
创建 GitHub Issue
```

飞书卡片交互需要后端接收卡片动作；而卡片交互服务端通常要快速响应，长任务应该异步执行，再更新卡片状态。飞书开放平台有卡片交互配置和长连接接收回调能力，适合你本地部署阶段避免公网回调地址的麻烦。([飞书开放平台][5])

## 3. 初赛建议：先不强依赖回调，但保留一个按钮小巧思

第一版可以只做通知。第二版加一个按钮：

```text
[查看 PR] [重跑测试] [转人工]
```

其中“查看 PR”只是链接按钮，不需要后端回调；“重跑测试 / 转人工”才需要回调。这样不会卡主线。

# 是否要监听 GitHub Issue？

建议做，但不要作为第一优先级。你的触发源优先级应是：

```text
P0：本地 demo 服务日志 traceback
P1：GitHub Issue，标题或 label 包含 bug
P2：pytest / GitHub Actions 失败日志
P3：Sentry / 线上监控，初赛可不接
```

因为初赛本地部署，没有云部署阶段，所以最稳的是：

```text
本地 FastAPI demo 服务
→ 写入 logs/app.log
→ Agent watcher 轮询日志
→ 解析 traceback
→ 进入修复流程
```

GitHub Issue 可以作为第二条入口：

```text
定时轮询 GitHub Issues
→ 找到 label=bug 且未处理
→ 读取 issue body 中的 traceback
→ 进入修复流程
```

这样比 webhook 更适合本地部署，不需要公网地址。

## 阶段 1：先做 demo 故障服务

目标：保证你有一个可控、可复现、可录屏的 bug 场景。

demo 服务建议用 FastAPI：

```text
GET /users/{user_id}/profile
```

预埋 bug：

```python
def get_user_profile(user_id: str):
    user = fake_db.get(user_id)
    return {
        "id": user["id"],
        "name": user["name"]
    }
```

当 `user_id` 不存在时触发：

```text
TypeError: 'NoneType' object is not subscriptable
```

或者：

```text
KeyError: 'name'
AttributeError: 'NoneType' object has no attribute 'id'
```

必须同时准备测试：

```text
tests/test_profile.py
```

失败测试示例：

```python
def test_missing_user_returns_404():
    response = client.get("/users/not-exist/profile")
    assert response.status_code == 404
```

这样 Agent 的目标很明确：补上空值判断，让测试通过。

## 阶段 2：实现日志监听与 Incident 归一化

目标：从 `logs/app.log` 里自动发现 traceback。

状态流：


Incident 数据结构：

```python
class Incident(BaseModel):

    repo_path: str
    error_type: str
```

fingerprint 用于去重：

## 阶段 4：实现 Agent 四工具闭环

目标：严格满足题目要求的 Tool Use。

四个工具不要做成虚的，要有真实函数：

```text
Read Log：读取 incident.traceback 和上下文日志
Read Code：根据 traceback 定位文件，读取相关代码片段
Run Test：在 worktree 里运行 pytest
Git Commit：提交修复分支并创建 PR
```

Agent 流程：

```text
1. read_log(incident_id)
2. read_code(file_hint, line_no)
3. call_doubao_repair()
4. apply_patch()
5. run_test()
6. if pass: git_commit_and_pr()
7. update_feishu_card()
8. update_bitable_record()
```

重点：**不要直接在主工作区改代码**。使用 `git worktree` 建隔离目录。Git 官方文档说明一个仓库可以支持多个 working tree，适合同时 checkout 不同分支；完成后可以 remove 掉 linked worktree。([Git][6])

分支命名：

```text
fix/inc-20260424-001-missing-user-guard
```

commit message：

```text
fix: handle missing user in profile endpoint

Incident: INC-20260424-001
Root cause: missing None guard when user is absent
Test: pytest demo_service/tests/test_profile.py -q
```

PR 标题：

```text
[AutoRepair] Fix missing user guard in profile endpoint
```

PR body：

```md
## Incident
INC-20260424-001

## Root Cause
...

## Fix
...

## Verification
- pytest demo_service/tests/test_profile.py -q

## Risk
Low. Only adds guard for missing user.
```

## 阶段 5：接入 GitHub Issue 入口

目标：让系统不仅能监控服务日志，也能处理“新 bug 提交”。

你可以做一个轮询器：

```text
每 30 秒查询 GitHub Issues
筛选：
- state=open
- label=bug

- 没有 autorepair:processing 标签
```

处理后给 issue 打标签：

```text
autorepair:processing

autorepair:human-required
```

这里不必做 GitHub webhook，因为你是本地部署。轮询更稳、更容易录屏。

Issue 入口示例：

```md

Return 404 instead of 500.
```

这条入口能体现比赛题目里“有新的 bug 提交时”的要求。

## 阶段 6：加小巧思，而不是加大功能

你需要的是“可行的小创新”，不是复杂系统。建议加 4 个小巧思：

### 小巧思 1：风险分级

LLM 输出：

```json
{
  "risk_level": "low",
  "need_human_review": true,
  "blocked_reason": ""
}
```

规则：

```text
只改业务代码 + 测试通过 → 自动建 PR
涉及 .env / infra / migrations / auth → 不提交，转人工
测试失败 → 不提交，转人工
```

### 小巧思 2：失败也有价值

如果修复失败，飞书卡片不要只写“失败”，而是写：

```text
我没有直接修复，因为测试仍然失败。
我已整理根因、相关文件、失败命令，请人工接手。
```

这会显得系统成熟，而不是“只能成功演示”。

### 小巧思 3：多维表格做修复知识库

每次修复成功，记录：

```text
错误类型
根因
修复方式
测试命令
PR 链接
```

答辩时可以说：**系统不仅修复单次 bug，还沉淀组织级故障知识。**

### 小巧思 4：双模型路由

```text
Doubao 2.0：复杂代码修复
Doubao 1.6：摘要、卡片文案、失败报告
```

这很适合你已有资源，也能体现“成本意识”和“工程落地”。
